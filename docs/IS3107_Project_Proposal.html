IS3107 Project Proposal

Title: Market Data Pipeline with Forecasting and Decision Support Dashboard

1. Problem Statement and Motivation

Financial market data is high-volume, noisy, and time-sensitive, while useful insights must be delivered consistently and quickly. This project addresses that challenge by building a reliable end-to-end system that ingests stock data, computes technical indicators, generates model forecasts, and publishes insights through an interactive dashboard.

The motivation is threefold: integrate data engineering and analytics tools in one workflow, produce interpretable indicators and model outputs beyond raw OHLCV data, and ensure operational reliability through orchestration and dependency-based triggers. The focus is not only model accuracy but also reproducibility, data freshness, and pipeline observability.

2. Planned Datasets and Data Sources

2.1 Primary Dataset: Market Price Data

  • Entity: daily equity time series (current focus: AAPL).
  • Fields: date/time, open, high, low, close, volume (OHLCV).
  • Cadence: daily ingestion with backfill support.
  • Storage: PostgreSQL table stock_prices.

2.2 Derived Dataset: Technical Features

  • Trend: SMA(20), SMA(50), EMA(12), EMA(26).
  • Momentum: MACD, MACD signal, RSI(14), return_3d, return_5d, daily_return.
  • Volatility: Bollinger upper/lower, Bollinger width, volatility_14.
  • Volume signals: volume_sma_20, volume_ratio_20.
  • Storage: PostgreSQL table stock_features.

2.3 Model Output Dataset

  • Fields: predicted_date, predicted_close, actual_close, model_version, created_at.
  • Storage: PostgreSQL table model_predictions.

2.4 Pipeline Metadata

Airflow operational metadata (e.g., DAG run states and timings) will be queried for reliability and freshness monitoring on the dashboard.

3. High-Level Pipeline Design and Tools

3.1 Logical Flow

  1. Ingestion: fetch daily/historical price data, validate schema, upsert to stock_prices.
  2. Feature Engineering: compute indicators/returns, write to stock_features, publish dataset-ready event.
  3. Forecasting: trigger prediction after features refresh, run LSTM inference, store outputs in model_predictions.
  4. Serving: Superset queries PostgreSQL and refreshes visual outputs.

3.2 Tools

  • Apache Airflow for orchestration (scheduled + dataset-triggered DAGs).
  • PostgreSQL for storage and analytics queries.
  • Docker Compose for reproducible local deployment.
  • Apache Superset for BI dashboards and operational monitoring.
  • LSTM model pipeline for prediction and versioned outputs.

4. Intended Outputs

4.1 Dashboard Outputs

  • Market panel: price trend with SMA/EMA, candlestick + volume, RSI, MACD, Bollinger bands.
  • Prediction panel: predicted vs actual close, error trend, MAE/RMSE by model version, rolling directional accuracy.
  • Pipeline panel: data freshness cards, DAG success/failure counts, run-duration trend.

4.2 Model and Engineering Outputs

  • Daily forecast values and version-tagged prediction records.
  • SQL query pack for dashboard datasets and charts.
  • Containerized setup and documented workflow for reproducibility.

5. Scope, Feasibility, and Anticipated Challenges

5.1 Scope

In scope: single-ticker end-to-end pipeline (AAPL), automated refresh, LSTM forecasting, and integrated dashboard.

Out of scope: production cloud HA deployment, intraday low-latency trading signals, and portfolio execution optimization.

5.2 Feasibility

The project is feasible in the semester timeframe because major infrastructure is already operational: containerized services, Airflow DAG orchestration, PostgreSQL schema for prices/features/predictions, and Superset integration path. Remaining work focuses on hardening, evaluation visibility, and dashboard polish.

5.3 Anticipated Challenges and Mitigation

  • Data quality gaps: handle nulls/missing days with validation and fallback logic in DAG tasks.
  • Model drift: maintain weekly retraining, versioning, and trend-based error monitoring.
  • Dependency reliability: enforce dataset-triggered downstream execution to avoid stale inputs.
  • Container dependency issues: lock setup in compose/dockerfiles and keep health-check diagnostics.
  • Stakeholder interpretability: pair forecasts with indicators and operational health context.

6. Deliverable Quality Criteria

  • Automated daily refresh from ingestion to dashboard.
  • Consistent prediction generation with version tracking.
  • Clear visualization for both market interpretation and pipeline observability.
  • Reproducible setup on teammate environments with minimal manual intervention.

7. Conclusion

This project delivers a realistic full-stack analytics product: ingestion, feature engineering, forecasting, and dashboard-based decision support. It balances technical depth and implementation feasibility, while providing a clear extension path to multi-ticker and cloud-native deployment.
